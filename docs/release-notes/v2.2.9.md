# v2.2.9

> 根治 v2.2.5 起的"升级后变成初始化页面"事故。受影响用户**直接升级 v2.2.9 即可自动恢复**，无需任何人工操作；**本地数据从来没有真正丢失**。

## 重点修复（P0）

### 数据"丢失"问题根因修复

v2.2.7 已经为 Docker 场景做过一次 hotfix，但只覆盖了一种镜像默认占位特征。**根因其实在代码层**，影响所有部署形态（Docker / Linux 二进制 / Windows / Magisk）。v2.2.9 做了根治。

**4 条独立缺陷叠加爆发**：

1. **`server/config/config.go::Load()` 漏解析 `database.path`**——v2.2.6 引入 `Data.Dir/ScriptsDir/LogDir` 相对→绝对转换时遗漏了 sqlite 数据库路径，导致 `database.path` 仍按 cwd 解析。这是跨所有部署形态的代码层根因。
2. **路径解析基准是 cwd 不稳定**——Docker `WORKDIR`、systemd `WorkingDirectory`、Windows 双击 vs cmd 启动、用户从其他目录运行二进制……cwd 行为不一致，相对路径解析结果飘忽。v2.2.9 起改成**以 `config.yaml` 所在目录为锚**，无论从哪里启动都稳定。
3. **`main.go` 硬编码 `config.Load("config.yaml")`**——没用已有的 `appboot.ResolveConfigPath()`，二进制部署 cwd 漂走就读不到 config。v2.2.9 统一走 `ResolveConfigPath`，并新增"可执行文件同目录的 `config.yaml`"作为查找候选。
4. **v2.2.7 entrypoint 检测过窄**——只识别硬编码的 `path: ./data/daidai.db` / `dir: ./data` 两条字符串，用户改过 config 但路径仍是任意相对形态的会漏检。v2.2.9 改为：**只要 `database.path` 或 `data.dir` 不是绝对路径，就重写为 `${DATA_DIR}/...`**。

### 数据兜底扫描（覆盖任意自定义挂载点）

entrypoint 每次启动会**两阶段扫描**整个容器内的 `daidai.db`：

1. **常见挂载点**：`/app/data`、`/app/Dumb-Panel`、`/data`、`/config`、`/opt/daidai`、`/app`
2. **浅 find 兜底**：从 `/` 起 4 层深度扫 `daidai.db`，自动发现任何用户自定义挂载点（如 `-v /host/x:/my-path`）

发现非空且与当前 `DATA_DIR` 不一致的残留库时，在容器日志里列出**所有候选 + 文件大小 + 修改时间 + 恢复命令**：

```
================================================================
检测到历史数据库残留（当前配置使用：/app/Dumb-Panel/daidai.db）：
  /app/data/daidai.db  (28672 字节, 修改时间 2026-05-15 09:30:12)
  /data/daidai.db      (1245184 字节, 修改时间 2026-05-20 18:42:55)

如其中某个是你的真实旧数据（v2.2.6 升级时被错位创建），执行恢复：
  1) 选定要恢复的源路径 SRC（推荐挑文件最大、修改时间最新的）
  2) docker exec <容器名> sh -c "cp -a SRC /app/Dumb-Panel/daidai.db; \
       cp -a SRC-shm /app/Dumb-Panel/daidai.db-shm 2>/dev/null; \
       cp -a SRC-wal /app/Dumb-Panel/daidai.db-wal 2>/dev/null"
  3) docker restart <容器名>

⚠️ 若残留只是 v2.2.6 错位生成的几 KB 空库，可忽略——直接用当前数据目录即可。
================================================================
```

**特别说明：用 `DATA_DIR` 自定义路径的用户**——常见的 NAS 部署是 `-v /mnt/nas/data:/data` + `DATA_DIR=/data`。v2.2.9 entrypoint 会基于你设置的 `DATA_DIR` 重写 config.yaml，旧数据自动找回。即使你没设 `DATA_DIR`（用了非默认挂载点但忘记同步环境变量），上述兜底扫描会把所有候选库列在日志里，给出准确的恢复命令——**不需要肉眼排查路径**。

## 其他修复

- **Dockerfile `ARG VERSION` 长期停留在 `2.0.4`** 的隐藏 bug 一并修正，构建出来的镜像版本号现在与实际版本一致。
- 启动日志新增 `config loaded: path=... db=... data_dir=...`，配置解析结果可观测，方便排障。
- 新增/补强单元测试覆盖：`Database.Path` 解析、cwd 与 config 目录不同时基于 config 目录解析、绝对路径不被改写。

## 升级路径

- **所有受 v2.2.6~v2.2.8 影响、面板进入"初始化"界面的 Docker 用户**：
  - 升级到 v2.2.9（`docker compose pull && docker compose up -d`，或等 watchtower 自动拉取）
  - 容器重启时 entrypoint 自动重写 `/app/config.yaml` 为绝对路径，旧数据立即可见
  - 如有残留库与当前数据目录不一致，日志里会直接给出恢复命令
- **仍在 v2.2.5 的用户**：直接升级到 v2.2.9
- **二进制 / Windows / Magisk 用户**：本次同步加固了路径解析，建议一并升级。即使你在 v2.2.5 之前没遇到问题，升级 v2.2.9 后会更稳（cwd 漂移不再影响）。

## 兼容性

- 已用绝对路径的 config.yaml（绝大多数已部署用户）：行为零变化。
- 用相对路径的 config.yaml：解析锚点从 cwd 改为 config 目录。语义更稳定，无副作用。
